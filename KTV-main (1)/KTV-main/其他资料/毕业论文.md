## 摘  要

随着互联网技术的快速发展和移动终端的普及，线上娱乐方式日益丰富。传统的线下KTV模式受限于场地和时间，难以满足用户随时随地进行社交娱乐的需求。本文设计并实现了一套名为"声域友"的线上社交KTV平台，旨在为用户提供通过浏览器和微信小程序进行在线K歌唱、实时聊天的社交娱乐体验。

系统采用前后端分离的B/S架构，后端基于ASP.NET Core 8.0框架，使用Dapper作为数据访问层，SQL Server作为数据库，通过RESTful API和SignalR实现实时通信；前端管理后台和用户端采用Vue 3 + TypeScript + Tailwind CSS技术栈，移动端采用微信小程序原生开发。系统实现了用户注册登录、虚拟房间管理、歌曲库管理、在线点歌排队、实时聊天、播放状态同步等核心功能。

本设计首先对系统进行了需求分析和概要设计，然后详细阐述了数据库设计、接口设计和各模块的详细设计与实现过程。系统测试结果表明，各功能模块运行稳定，能够满足线上社交KTV的基本业务需求，具有一定的实用价值。

**关键词**：线上KTV；社交娱乐；Vue 3；ASP.NET Core；微信小程序

## Abstract

With the rapid development of Internet technology and the popularization of mobile terminals, online entertainment methods have become increasingly diverse. The traditional offline KTV model is limited by venue and time, making it difficult to meet users' needs for social entertainment anytime and anywhere. This paper designs and implements an online social KTV platform named "Hydro-Sonic Canvas", aiming to provide users with social entertainment experiences of online karaoke singing and real-time chatting through browsers and WeChat Mini Programs.

The system adopts a front-end and back-end separated B/S architecture. The back-end is based on the ASP.NET Core 8.0 framework, using Dapper as the data access layer and SQL Server as the database, achieving real-time communication through RESTful API and SignalR. The front-end management console and user terminal use the Vue 3 + TypeScript + Tailwind CSS technology stack, while the mobile terminal adopts native WeChat Mini Program development. The system implements core functions including user registration and login, virtual room management, song library management, online song ordering and queuing, real-time chat, and playback state synchronization.

This design first conducts requirements analysis and preliminary design of the system, then elaborates on the database design, interface design, and the detailed design and implementation process of each module. System test results show that each functional module runs stably and can meet the basic business requirements of online social KTV, possessing practical value.

**Key words**: Online KTV; Social Entertainment; Vue 3; ASP.NET Core; WeChat Mini Program

## 1 绪论

### 1.1 研究背景

近年来，随着移动互联网技术的飞速发展和智能终端设备的广泛普及，人们的娱乐方式发生了深刻变革。传统的线下KTV作为重要的社交娱乐场所，长期以来深受广大用户的喜爱。然而，线下KTV模式存在诸多限制：首先，用户需要前往固定场所才能享受KTV服务，受到地理位置的约束；其次，KTV包厢通常需要提前预约，且营业时间有限，无法满足用户随时随地的娱乐需求；此外，线下KTV的消费成本相对较高，对于学生群体和年轻用户来说存在一定经济压力。

与此同时，社交网络和在线娱乐产业蓬勃发展，用户对于线上社交娱乐的需求日益增长。特别是在后疫情时代，远程社交和在线娱乐成为人们重要的生活方式。现有的在线音乐平台如网易云音乐、QQ音乐等虽然提供了丰富的音乐资源，但缺乏实时互动和社交功能，无法满足用户与朋友一起在线K歌的需求。因此，开发一套集在线K歌、实时社交、虚拟房间于一体的线上社交KTV平台，具有重要的现实意义和应用价值。

### 1.2 研究目的

本设计旨在开发一套名为"声域友"的线上社交KTV平台，主要研究目的包括：

（1）设计并实现一个功能完善的线上社交KTV系统，涵盖用户管理、歌曲管理、房间管理、在线点歌、实时聊天等核心功能，为用户提供便捷的在线K歌体验。

（2）采用前后端分离的现代化开发架构，前端使用Vue 3 + TypeScript技术栈，后端使用ASP.NET Core 8.0框架，实现系统的高内聚、低耦合，提高代码的可维护性和可扩展性。

（3）实现微信小程序端，使用户可以通过移动设备随时随地访问平台，扩大系统的使用场景和用户覆盖面。

（4）通过实时通信技术实现播放状态同步和即时聊天功能，提升用户的社交互动体验。

（5）在完成系统开发的基础上，总结Web应用开发的技术方法和实践经验，为后续类似项目提供参考。

### 1.3 国内外研究现状

在在线KTV领域，国内外已有一些成熟的产品和研究成果。在国外，Smule、Karaoke等应用较早进入在线K歌市场，提供了丰富的歌曲库和社交功能。Smule应用支持用户录制和分享演唱视频，具有较强的社交属性，但主要面向海外市场，国内用户使用较少。

在国内，在线K歌市场发展迅速，唱吧、全民K歌等产品占据了较大的市场份额。唱吧作为国内较早的在线K歌平台，提供了实时对唱、合唱、PK等功能，用户规模庞大。全民K歌依托腾讯的社交生态，实现了与微信、QQ的深度整合，用户可以方便地分享演唱作品到社交平台。这些产品主要面向移动端用户，以手机APP为主要载体。

在技术实现方面，现有的在线K歌系统大多采用微服务架构，使用WebSocket实现实时通信，采用音频处理技术实现音效调节和伴奏分离。然而，这些系统普遍存在开发成本高、技术复杂度大的特点，对于中小型团队和学术研究项目来说，实现难度较大。

本设计在充分调研现有产品的基础上，结合实际需求和技术条件，采用轻量级的技术方案实现线上社交KTV平台。系统不追求复杂的功能和华丽的特效，而是聚焦于核心的K歌社交功能，注重系统的实用性和可实现性，为类似项目提供一个可行的技术参考方案。

### 1.4 研究内容

本设计的主要研究内容包括：

（1）系统需求分析：对线上社交KTV平台进行详细的需求调研和分析，明确系统的功能需求和非功能需求。

（2）系统架构设计：设计合理的系统整体架构，包括前后端分离架构、后端分层架构、数据库架构等。

（3）数据库设计：根据业务需求设计数据库表结构，建立合理的表关系和索引，确保数据存储的完整性和查询效率。

（4）后端开发：基于ASP.NET Core 8.0框架开发RESTful API接口，实现用户认证、歌曲管理、房间管理、播放控制等后端业务逻辑。

（5）前端开发：使用Vue 3 + TypeScript开发管理后台和用户端Web应用，实现响应式界面和良好的用户体验。

（6）小程序开发：使用微信小程序原生框架开发移动端应用，实现核心功能的移动化访问。

（7）系统测试：对系统进行功能测试和性能测试，验证系统的正确性和稳定性。

## 2 相关技术介绍

### 2.1 前端技术

#### 2.1.1 Vue 3

Vue 3是一个用于构建用户界面的渐进式JavaScript框架，由尤雨溪开发并于2020年正式发布。相比Vue 2，Vue 3在性能、开发体验和TypeScript支持等方面都有显著提升。Vue 3引入了Composition API，允许开发者使用函数的方式组织组件逻辑，提高了代码的可复用性和可维护性。同时，Vue 3采用了Proxy响应式系统替代了Vue 2的Object.defineProperty，提供了更好的响应式性能和更全面的数据拦截能力。

本系统前端采用Vue 3作为核心框架，结合Composition API和`<script setup>`语法糖，实现了组件逻辑的清晰组织和高效开发。

#### 2.1.2 TypeScript

TypeScript是JavaScript的一个超集，由微软开发并维护。TypeScript在JavaScript的基础上添加了静态类型检查，支持接口、泛型、枚举等高级类型特性，能够在编译阶段发现潜在的错误，提高代码的可靠性和可维护性。

本系统前端全面采用TypeScript进行开发，通过定义接口和类型，实现了前后端数据的一致性约束，减少了运行时错误的发生。

#### 2.1.3 Tailwind CSS

Tailwind CSS是一个功能优先的CSS框架，提供了大量的原子化CSS类，开发者可以通过组合这些类快速构建自定义界面。相比传统的CSS框架如Bootstrap，Tailwind CSS不提供预定义的组件样式，而是给予开发者更大的自由度来设计独特的界面风格。

本系统使用Tailwind CSS 3.x作为样式解决方案，通过其灵活的配置系统和响应式设计支持，实现了美观且一致的用户界面。

#### 2.1.4 Element Plus

Element Plus是基于Vue 3的企业级UI组件库，提供了丰富的表单、表格、对话框、通知等常用组件。Element Plus遵循Material Design设计规范，组件风格统一、交互规范，能够显著提高开发效率。

本系统管理后台使用Element Plus作为UI组件库，利用其成熟的组件实现数据表格、表单验证、分页等功能。

### 2.2 后端技术

#### 2.2.1 ASP.NET Core 8.0

ASP.NET Core是微软开发的跨平台、高性能的Web应用框架，支持构建RESTful API、Web应用和实时应用。ASP.NET Core 8.0是该框架的最新版本，提供了更好的性能、简化的配置和增强的安全特性。ASP.NET Core采用依赖注入作为内置功能，支持中间件管道模式，具有良好的可扩展性。

本系统后端采用ASP.NET Core 8.0框架，利用其内置的依赖注入、中间件、认证授权等功能，构建了清晰的分层架构。

#### 2.2.2 Dapper

Dapper是一个轻量级的ORM（对象关系映射）框架，由Stack Overflow团队开发。相比Entity Framework等全功能ORM，Dapper更加轻量和高效，它通过扩展IDbConnection接口，提供了简单的对象映射功能，同时保留了原生SQL的灵活性。

本系统使用Dapper作为数据访问层，通过Repository模式封装数据库操作，既保证了开发效率，又保留了对SQL的完全控制能力。

#### 2.2.3 JWT认证

JWT（JSON Web Token）是一种开放标准（RFC 7519），用于在网络应用环境间安全地传递声明信息。JWT由三部分组成：头部（Header）、载荷（Payload）和签名（Signature），具有自包含、无状态、易于分布式扩展等特点。

本系统采用JWT作为用户认证方案，用户登录成功后服务器返回JWT令牌，客户端在后续请求中携带令牌进行身份验证，实现了无状态的认证机制。

### 2.3 数据库技术

SQL Server是微软开发的关系型数据库管理系统，具有强大的数据处理能力、完善的事务支持和丰富的管理工具。SQL Server支持T-SQL查询语言，提供了存储过程、触发器、视图等数据库对象，适合企业级应用的数据存储需求。

本系统使用SQL Server作为数据存储，利用其成熟的事务处理机制和索引优化能力，确保数据的一致性和查询性能。

### 2.4 实时通信技术

#### 2.4.1 SignalR

SignalR是ASP.NET Core内置的实时通信库，支持WebSocket、Server-Sent Events和长轮询等多种传输方式，能够根据客户端和服务端的能力自动选择最佳的传输方式。SignalR提供了Hub抽象，允许客户端和服务端之间进行双向通信。

本系统使用SignalR实现房间内的实时事件推送，包括用户加入退出、播放状态变化、队列更新等事件的实时通知。

#### 2.4.2 REST轮询

REST轮询是一种简单可靠的实时通信方案，客户端定期向服务器发送请求获取最新数据。虽然轮询相比WebSocket有一定的延迟，但其实现简单、兼容性好，适合对实时性要求不高的场景。

本系统在聊天消息和部分房间状态同步场景中采用2秒间隔的REST轮询方案，与SignalR形成互补，兼顾了实时性和实现复杂度。

## 3 系统需求分析

### 3.1 系统概述

声域友线上社交KTV平台是一套基于Web的在线K歌社交系统，用户可以通过浏览器或微信小程序创建或加入虚拟房间，在房间内与朋友一起点歌、排队、实时聊天，享受线上KTV体验。系统分为三个主要部分：PC端管理后台、PC端用户前端和微信小程序移动端。

管理后台面向系统管理员，提供房间监控、歌曲管理、用户管理、反馈处理等运营管理功能。用户前端面向普通用户，提供歌曲探索、房间加入、在线点歌、实时聊天等核心功能。微信小程序提供移动端的访问入口，使用户可以随时随地使用平台服务。

### 3.2 功能需求分析

#### 3.2.1 用户管理需求

系统需要提供完整的用户管理功能，包括：

（1）用户注册：支持通过用户名、手机号或邮箱进行注册，注册时需要设置密码和昵称。

（2）用户登录：支持通过用户名、手机号或邮箱配合密码进行登录，登录成功后获取JWT令牌。

（3）用户信息管理：用户可以查看和编辑个人资料，包括昵称、头像、联系方式等。

（4）密码管理：用户可以修改登录密码，修改前需要验证原密码。

（5）管理员管理：管理员可以查看用户列表、创建用户、禁用/启用用户、重置密码等。

#### 3.2.2 歌曲管理需求

系统需要提供完善的歌曲库管理功能：

（1）歌曲浏览：用户可以按流派分类浏览歌曲，支持关键词搜索。

（2）歌曲详情：用户可以查看歌曲的详细信息，包括歌手、流派、时长等。

（3）歌曲上传：管理员可以上传新的歌曲，包括音频文件、封面图片和歌词文件。

（4）歌曲编辑：管理员可以编辑歌曲信息，包括歌名、歌手、流派等。

（5）歌曲删除：管理员可以删除不需要的歌曲。

（6）播放统计：系统记录每首歌曲的播放次数，支持按播放量排序。

#### 3.2.3 房间管理需求

虚拟房间是系统的核心概念，相关需求包括：

（1）房间创建：用户可以申请创建虚拟房间，需要经过管理员审批。

（2）房间加入：用户可以通过输入房间码加入已存在的房间。

（3）房间退出：用户可以主动退出当前所在的房间。

（4）房间监控：管理员可以查看所有活跃房间的状态，包括房间码、在线人数、创建时间等。

（5）房间关闭：管理员可以手动关闭房间，关闭后房间内所有用户将被踢出。

（6）空闲关闭：房间内无人时，系统会在5分钟后自动关闭房间。

#### 3.2.4 在线点歌需求

在线点歌是系统的核心功能，相关需求包括：

（1）点歌：房间内用户可以从歌曲库中选择歌曲加入播放队列。

（2）队列管理：用户可以调整自己点播歌曲的顺序，也可以删除自己的点播记录。

（3）播放控制：当前歌曲的点歌人可以控制播放/暂停、切歌、调整播放模式。

（4）播放模式：支持顺序播放、列表循环、单曲循环、随机播放四种模式。

（5）播放同步：房间内所有用户的播放状态保持同步。

#### 3.2.5 实时聊天需求

实时聊天是增强社交体验的重要功能：

（1）消息发送：房间内用户可以发送文字聊天消息。

（2）消息展示：聊天消息实时显示在房间界面中，支持自动滚动。

（3）系统消息：用户加入退出房间时自动发送系统通知消息。

（4）消息限制：单条消息最长200字符，支持敏感词过滤。

#### 3.2.6 收藏功能需求

用户可以收藏喜欢的歌曲，方便后续查找和点播：

（1）添加收藏：用户可以将歌曲添加到收藏列表。

（2）取消收藏：用户可以取消已收藏的歌曲。

（3）收藏列表：用户可以查看自己的收藏歌曲列表。

#### 3.2.7 排行榜需求

系统需要提供歌曲排行榜功能：

（1）日榜：展示当天播放量最高的歌曲排行。

（2）周榜：展示本周播放量最高的歌曲排行。

### 3.3 非功能需求分析

#### 3.3.1 性能需求

（1）页面首次加载时间应在3秒以内。

（2）API接口响应时间应在500毫秒以内。

## 4 系统设计

### 4.1 系统架构设计

#### 4.1.1 整体架构

系统采用前后端分离的B/S（Browser/Server）架构，整体分为表现层、业务逻辑层和数据层三个层次。

表现层包括三个客户端应用：PC端管理后台、PC端用户前端和微信小程序移动端。管理后台和用户前端基于Vue 3框架开发，运行在用户浏览器中；微信小程序基于微信原生框架开发，运行在微信客户端中。

业务逻辑层由ASP.NET Core 8.0后端服务提供，采用Repository-Service-Controller三层架构模式。Controller层负责接收和响应HTTP请求；Service层负责业务逻辑处理；Repository层负责数据访问操作。

数据层使用SQL Server关系型数据库，存储系统的所有持久化数据。

#### 4.1.2 后端架构

后端采用经典的三层架构设计：

（1）Controller层：包含15个API控制器，负责接收客户端请求，调用Service层处理业务逻辑，并返回响应结果。控制器通过依赖注入获取Service实例。

（2）Service层：包含7个业务服务和2个单例服务，负责核心业务逻辑的处理。业务服务处理常规的CRUD操作；单例服务负责播放状态管理和WebSocket通知。

（3）Repository层：包含12个Repository接口和实现，封装数据库操作。Repository通过Dapper执行SQL语句，实现数据的增删改查。

此外，后端还包含以下辅助组件：

- 中间件：全局异常处理、操作日志记录、用户活跃时间更新。
- SignalR Hub：处理实时通信，包括房间加入退出、状态同步等。
- DTO层：定义数据传输对象，实现层间数据的解耦。

#### 4.1.3 前端架构

前端管理后台和用户端采用相似的架构设计：

（1）路由层：使用Vue Router 4管理页面路由，通过路由守卫实现登录验证。

（2）状态层：使用Pinia进行全局状态管理，包括认证状态、播放器状态等。

（3）接口层：使用Axios封装HTTP请求，通过拦截器实现Token附加和错误处理。

（4）视图层：使用Vue 3 Composition API开发页面组件，Element Plus提供UI组件支持。

（5）样式层：使用Tailwind CSS实现响应式布局和自定义样式。

#### 5.3.2 播放器实现

播放器基于HTML5 Audio API实现，使用Pinia进行状态管理：

```typescript
export const usePlayerStore = defineStore('player', () => {
  const audio = new Audio()
  const queue = ref<PlayerTrack[]>([])
  const currentIndex = ref(-1)
  const isPlaying = ref(false)
  const currentTime = ref(0)
  const duration = ref(0)
  const playMode = ref<'off' | 'repeat-all' | 'repeat-one' | 'shuffle'>('off')

  function play() {
    audio.play().catch(() => {})
  }

  function pause() {
    audio.pause()
  }

  function seek(time: number) {
    audio.currentTime = time
  }

  // 同步远程播放状态
  function applyRemoteState(state: any) {
    if (state.isPlaying && audio.paused) play()
    else if (!state.isPlaying && !audio.paused) pause()
    if (Math.abs(audio.currentTime - state.currentTime) > 2) {
      seek(state.currentTime)
    }
  }

  return { queue, currentIndex, isPlaying, currentTime, duration, playMode, play, pause, seek, applyRemoteState }
})
```

#### 5.4.3 API封装

小程序通过封装的API模块与后端通信：

```javascript
// utils/api.js
const API_BASE = 'http://localhost:5276/api'

function request(url, method = 'GET', data = {}) {
  return new Promise((resolve, reject) => {
    const token = wx.getStorageSync('token')
    wx.request({
      url: API_BASE + url,
      method: method,
      data: data,
      header: {
        'Content-Type': 'application/json',
        'Authorization': token ? `Bearer ${token}` : ''
      },
      success: (res) => {
        if (res.statusCode === 200) {
          resolve(res.data)
        } else if (res.statusCode === 401) {
          wx.removeStorageSync('token')
          wx.redirectTo({ url: '/pages/index/index' })
        }
      },
      fail: reject
    })
  })
}
```

## 7 总结与展望

### 7.1 总结

本设计完成了声域友线上社交KTV平台的开发工作，实现了系统的所有预期功能。主要工作和成果包括：

（1）完成了系统的需求分析和架构设计，确定了前后端分离的技术方案和分层架构模式。

（2）完成了数据库设计，设计了12张数据表，建立了合理的表关系和索引。

（3）基于ASP.NET Core 8.0开发了后端服务，实现了62个RESTful API接口，覆盖了系统的全部业务功能。

（4）基于Vue 3 + TypeScript开发了PC端管理后台和用户前端，实现了响应式界面和良好的用户体验。

（5）基于微信小程序原生框架开发了移动端应用，实现了核心功能的移动化访问。

（6）通过SignalR和REST轮询实现了实时通信功能，支持播放状态同步和即时聊天。

（7）对系统进行了全面的功能测试和性能测试，验证了系统的正确性和稳定性。

通过本次毕业设计，深入实践了Web应用开发的完整流程，掌握了Vue 3、ASP.NET Core、SQL Server等主流技术的使用方法，提高了软件工程实践能力和问题解决能力。

### 7.2 展望

虽然系统已实现核心功能，但仍有一些方面可以进一步完善：

（1）实时通信优化：将聊天消息的REST轮询改为WebSocket长连接，进一步降低延迟。

（2）音频处理功能：增加音效调节、伴奏分离等音频处理功能，提升K歌体验。

（3）社交功能增强：增加好友系统、合唱功能、演唱作品分享等社交功能。

（4）性能优化：引入缓存机制、数据库读写分离等技术，提高系统并发能力。

（5）移动端适配：对Web端进行移动端响应式适配，提供更好的移动体验。

（6）部署上线：将系统部署到云服务器，配置域名和SSL证书，正式对外提供服务。
