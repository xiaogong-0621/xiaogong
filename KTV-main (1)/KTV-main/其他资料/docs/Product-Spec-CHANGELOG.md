# 声域友 - 产品需求变更记录

## [2.0.8] - 2026-05-21

### 变更类型：新增（播放列表固定正在播放位置）+ 修改（歌词页播放列表同步）

### 新增
- 歌词页全屏覆盖：漏斗式歌词滚动（当前行放大加亮，远处缩放渐隐）+ 播放列表侧栏（拖拽排序、收藏、自动滚屏）
- 播放模式切换：顺序播放 → 列表循环 → 单曲循环 → 随机播放
- 音量控制：歌词页左侧悬浮展开滑块
- LRC 歌词文件加载：从数据库 lrcUrl 字段获取歌词文件，解析 [mm:ss.xx] 格式
- FLAC 音频格式支持：注册 MIME 类型 audio/flac，Kestrel 请求体上限提升至 100MB
- 批量排序 API：`POST /api/room/queue/reorder-batch`，事务内更新 SortOrder
- 播放列表"正在播放"固定卡片：当前播放歌曲始终在 index 0，UI 分区为"正在播放"+ "接下来"
- 点击播放列表歌曲自动移到正在播放位置

### 修改
- 播放列表统一数据源：CurrentRoomView 去掉本地 queue 数组，改用 player.queue 派生的 displayQueue computed，消除双队列不同步
- 拖拽排序持久化到服务器：拖拽完成后调 reorderBatch API，轮询不会覆盖
- 拖拽禁止操作：当前播放歌曲不可拖动，其他歌曲不可拖到正在播放位置
- 切歌逻辑适配 index 0 固定：playNext 将下一首移到 index 0，playPrev 重启当前歌曲
- 歌词页播放列表与当前房间页同步：统一"正在播放"+ "接下来"分区结构
- CORS 中间件顺序：UseCors 移到 UseStaticFiles 之前，解决静态文件跨域问题
- SongRepository SQL：INSERT/UPDATE 语句补全 LrcUrl 列

### 原因
- 歌词页需要播放列表侧栏方便切歌和排序
- 播放列表"正在播放"位置固定更符合 KTV 点歌体验
- 双队列不同步导致点击播放后 UI 不更新

## [2.0.7] - 2026-05-19

### 变更类型：新增（全屏歌词页）+ 修改（播放器重写为悬浮胶囊）

### 新增
- 全屏歌词页：点击胶囊播放器的圆形封面从底部滑入，封面模糊放大做背景 + 毛玻璃遮罩，中间大封面 256px + 歌名/歌手 + 歌词占位区域，底部粗进度条 + 实心圆播放按钮
- 歌词页关闭：顶部收起按钮 + Escape 键
- 歌词页播放控制：上一首/下一首按钮（灰化首尾），进度条可拖拽，进度与胶囊播放器实时同步

### 修改
- 底部播放器栏：从全宽横栏（`h-24`，`rounded-t-3rem`）重写为底部居中悬浮胶囊（`max-width: 560px`，`rounded-full`），圆形封面 56px + 歌名/歌手 + 内嵌进度条 + 三键控制 + 时间
- 封面交互：悬浮时放大 + 显示上箭头图标提示可点击，无歌曲播放时光标非指针、不响应点击
- 播放按钮：从 `play_circle`/`pause_circle` 图标改为 `play_circle` 填充样式（`FILL 1`）
- 进度条：从播放器栏顶部 1px 线改为内嵌在歌名下方 1px 渐变条，悬浮显示拖拽手柄
- 主内容区底部间距：从 `pb-32` 微调为 `pb-28`，适配更小的胶囊播放器

### 删除
- 播放器栏中的音量滑块（当前版本精简，后续按需恢复）
- 播放器栏顶部的渐变色装饰线

### 原因
- 全宽播放器栏占用空间大，悬浮胶囊更轻量现代
- 歌词页提供沉浸式听歌体验，封面做背景增强视觉层次
- 未来歌词功能上线后，歌词页将成为核心交互页面

- 单一渐变背景视觉单调，多层浮动色块更有空间感和活力
- 封面悬浮放大效果视觉突兀且遮挡其他内容，整行悬浮预览更统一自然
- 侧边栏收起可释放更多内容区空间，适配不同使用场景

## [2.0.2] - 2026-05-16

### 变更类型：修改（聊天技术方案调整）

### 修改
- 实时聊天：技术方案从 SignalR 改为 REST API + 轮询（每 2 秒）
- 聊天消息：从不持久化改为持久化到 ChatMessages 表，退出房间后保留
- 聊天 REST API：新增 `POST /api/chat/send` 和 `GET /api/chat/messages` 接口
- 数据库：新增 ChatMessages 表（RoomId, UserId, Nickname, Content, CreatedAt）
- SignalR Hub：移除 SendMessage 方法，聊天不再经 SignalR 传输
- 当前房间页：前端移除 SignalR 依赖，改为 REST 轮询获取聊天消息和播放队列
- 版本号从 2.0.1 更新为 2.0.2

### 原因
- SignalR Hub 的 `JoinRoom` 和 `SendMessage` 方法在服务端无法正常执行（WebSocket 框架层问题，无服务端日志输出）
- 轮询方案简单可靠，KTV 聊天非高频场景，2 秒延迟可接受

## [2.0.0] - 2026-05-15

### 变更类型：重构（产品方向转型）

### 新增
- 注册功能：支持手机号、邮箱、用户名+密码三种注册方式，无需验证码
- 登录后房间选择：登录成功后弹出二级窗口，用户可选择输入房间码加入或申请开房间
- 实时聊天：当前房间页新增实时聊天区域，消息实时同步给房间内所有用户，优先使用 SignalR 实现
- 用户反馈：探索歌曲页新增反馈浮动按钮，用户可提交"请求添加歌曲""歌曲信息纠错""其他建议"三种反馈
- 反馈管理（管理端）：新增反馈管理页面，管理员可查看和处理用户反馈；右上角通知图标接入待处理反馈列表，支持直接标记已处理
- 房间监控 - 开房申请审批：管理员在房间监控页审批用户的开房申请，通过后系统分配房间并生成 6 位房间码
- 房间生命周期管理：空房间（所有用户退出）进入 5 分钟倒计时，到时自动关闭，房间码作废；有人加入则重置倒计时

### 修改
- 产品定位：从"线下KTV点歌管理系统"转型为"线上社交KTV平台"
- 房间管理 → 房间监控：从管理物理包厢改为监控虚拟房间，包含活跃房间列表和待审批列表
- 仪表盘：统计指标从收入相关改为在线房间数、在线用户数、今日创建房间数、总注册用户数
- 账户管理：移除余额系统和 VIP 标签，保留禁用/启用功能；禁用时如用户在房间中则自动退出
- 系统设置：移除定价设置（时薪、包厢类型倍率、节假日动态定价），从六个配置区块精简为四个（管理员账号、基本信息、安全设置、操作日志）
- 操作日志：记录的操作类型从订单相关改为房间和反馈相关
- 安全设置：敏感操作验证项从订单/余额相关改为禁用用户/关闭房间/修改设置相关
- 当前房间：新增实时聊天功能，置顶和删除操作限制为仅对自己点播的歌曲生效；退出房间时自动清除该用户点播的歌曲
- 目标用户：管理端从"KTV运营人员"改为"系统管理员"，Web 端从"KTV顾客"改为"线上KTV用户"
- 技术栈：新增 SignalR（实时通信），后端框架说明从纯 RESTful API 扩展为 RESTful API + SignalR Hub
- AI 增强建议：移除"营业数据洞察"（依赖订单数据，已移除），新增"聊天消息智能过滤"

### 删除
- 订单管理：整个模块移除（产品转型为社交KTV平台，不再有消费结算流程）
- 余额系统：用户余额、充值/扣款功能移除
- VIP 系统：VIP 用户标签、VIP 包厢权限、VIP 状态切换功能移除
- 定价设置：包厢时薪、包厢类型倍率、节假日动态定价、节假日日期管理全部移除
- 代码审查决策记录：标记涉及订单的决策为"已过时"

### 功能调整
- Web 端新增注册功能优先级：高（P0）
- Web 端房间加入/申请功能优先级：高（P0）
- 管理端房间监控（含审批）优先级：高（P0），原房间管理优先级保持不变
- 反馈管理优先级：中（P2）

### 技术栈调整
- 新增 SignalR：用于房间内实时聊天和播放队列同步
- 新增 SignalR Hub：后端实时通信组件

### 影响范围
- 数据库：需要新增反馈表、房间表（虚拟房间，替代原包厢概念），移除订单表、余额相关字段
- 后端 API：移除订单相关 Controller，新增房间管理（创建/加入/退出/审批）、反馈、聊天相关接口
- 前端：新增注册页面、房间选择弹窗、聊天组件、反馈弹窗；移除所有订单相关 UI
- 管理端：移除订单管理页面，新增反馈管理页面；房间管理改为房间监控

### 兼容性说明
- 不兼容：这是产品方向的根本转型，从 v1.x 到 v2.0.0 不保留向后兼容
- 数据库需要重新设计，大部分表结构需要调整
- 前端和后端代码需要大幅重构
