# 实时数据同步机制 —— 以"点歌同步"为例

## 问题

用户 A 在房间点了一首歌，用户 B 要能**实时看到**播放队列更新。怎么做到的？

---

## 方案选型

| 方案 | 原理 | 实时性 | 复杂度 |
|------|------|--------|--------|
| **WebSocket（SignalR）** | 服务端主动推送 | 毫秒级 | 高 |
| **轮询（Polling）** | 前端定时请求 | 取决于间隔 | 低 |
| **长轮询（Long Polling）** | 请求挂起直到有新数据 | 秒级 | 中 |

本项目最终采用**轮询方案**，间隔 2 秒。SignalR 在本项目中因框架层问题无法正常执行 Hub 方法，详见项目记忆。

---

## 完整数据流

### 用户 A 点歌

```
用户 A 点击"点歌"
    ↓
前端 useSongOrder.ts → roomApi.orderSong(songId, 0)
    ↓
HTTP POST /api/room/queue { songId: 42 }
    ↓
后端 RoomController.AddToQueue()
    ├── 查 RoomUsers 表 → 用户 A 在哪个房间（roomId = 4）
    ├── 查 Songs 表 → 歌曲是否存在且已上架
    ├── 查 PlayQueue 表 → 是否已重复点歌
    └── INSERT INTO PlayQueue (RoomId, SongId, OrderedByUserId, SortOrder, Status)
        VALUES (4, 42, 25, 3, 'queued')
    ↓
返回 { id: 108 } 给用户 A 的前端
```

### 用户 B 看到更新

```
用户 B 的 CurrentRoomView 正在运行
    ↓
setInterval 每 2 秒执行一次 loadQueue()
    ↓
前端 roomApi.getQueue()
    ↓
HTTP GET /api/room/queue
    ↓
后端 RoomController.GetQueue()
    ├── 查 RoomUsers 表 → 用户 B 在房间 4
    └── SELECT * FROM PlayQueue WHERE RoomId = 4 AND Status = 'queued'
        ORDER BY SortOrder
    ↓
返回最新的播放队列（包含用户 A 刚点的歌）
    ↓
前端 queue.value = data → Vue 响应式更新 → UI 刷新
```

**关键点**：用户 A 点歌后，数据库 PlayQueue 表多了一条记录。用户 B 的下一次轮询（最多 2 秒后）会查到这条新记录。

## 聊天同步：同样的模式

聊天和点歌**完全一样的同步机制**，只是写入和读取的 API 不同。

### 数据流

```
用户 A 输入"hello"点击发送
    ↓
前端 chatApi.sendMessage(roomId, "hello")
    ↓
HTTP POST /api/chat/send { roomId: 4, message: "hello" }
    ↓
后端 ChatController.SendMessage()
    ├── 验证消息非空且 ≤ 200 字符
    ├── 敏感词过滤（"操" → "**"）
    └── INSERT INTO ChatMessages (RoomId, UserId, Nickname, Content, CreatedAt)
    ↓
返回 { id: 88 } 给用户 A

用户 B 的下一次轮询（最多 2 秒后）
    ↓
HTTP GET /api/chat/messages?roomId=4
    ↓
后端 ChatController.GetMessages()
    └── SELECT TOP 50 * FROM ChatMessages WHERE RoomId = 4 ORDER BY CreatedAt DESC
    ↓
前端 chatMessages.value = data → Vue 响应式更新 → 聊天列表刷新
```

### 后端：写入

[HttpPost("send")]
public async Task<IActionResult> SendMessage([FromBody] SendMessageRequest request)
{
    if (string.IsNullOrWhiteSpace(request.Message))
        return BadRequest(new { message = "消息不能为空" });
    if (request.Message.Length > 200)
        return BadRequest(new { message = "消息长度不能超过200字符" });

    var userId = GetUserId();
    var nickname = GetNickname();
    var id = await _chatRepo.CreateAsync(request.RoomId, userId, nickname, request.Message);
    return Ok(new { id });
}
```

### 后端：读取

[ChatController.cs:38](../backend/Controllers/ChatController.cs#L38)

[HttpGet("messages")]
public async Task<IActionResult> GetMessages([FromQuery] int roomId, [FromQuery] int limit = 50)
{
    var messages = await _chatRepo.GetMessagesAsync(roomId, limit);
    // 返回最近 50 条，按时间正序
    var result = messages.Select(m => new ChatMessageResponse(
        m.Id, m.Nickname, m.Content, m.CreatedAt.ToLocalTime().ToString("HH:mm")
    ));
    return Ok(result);
}
```

### 对比：点歌 vs 聊天

| 维度 | 点歌 | 聊天 |
|------|------|------|
| 写入 API | `POST /api/room/queue` | `POST /api/chat/send` |
| 读取 API | `GET /api/room/queue` | `GET /api/chat/messages` |
| 存储表 | PlayQueue | ChatMessages |
| 持久化 | 是（歌曲播放后变 removed） | 是（永久保留） |
| 读取后清空 | 否 | 否（每次返回最近 50 条） |
| 服务端处理 | 去重校验 | 敏感词过滤 |
| 轮询方式 | 同一个 setInterval，每 2 秒 | 同一个 setInterval，每 2 秒 |
| 写入后前端更新 | 靠下次轮询（≤2秒） | 靠下次轮询（≤2秒） |

## 优缺点

### 优点
- 实现简单，不需要 WebSocket 或 SignalR
- 后端无状态，每个请求独立处理
- 天然支持断线重连（下次轮询自动恢复）

### 缺点
- 2 秒延迟（用户 B 最多等 2 秒才能看到 A 的操作）
- 每个用户每秒产生 0.5 次无用请求（空轮询）
- 用户越多，服务器压力越大

### 适用场景
- 非高频交互（聊天、播放队列）
- 用户量小（课程项目）
- 对实时性要求不严格（差 1-2 秒可接受）

### 不适用场景
- 大规模并发（1000+ 用户同时在线）
- 需要毫秒级同步（游戏、股票行情）
- 移动端流量敏感（轮询耗电）

## 替代方案：SignalR（项目曾尝试）

产品文档中原本设计使用 SignalR 实现，核心思路：

```
用户 A 点歌 → 后端 INSERT PlayQueue
            → 后端调用 Clients.Group("room-4").SendAsync("QueueUpdated")
            → 用户 B 的前端 connection.on('QueueUpdated', () => loadQueue())
            → 用户 B 立即看到更新（毫秒级）
```

因 SignalR Hub 方法在服务端无法执行（无日志输出、无异常信息），最终改为轮询。根因未定位。
